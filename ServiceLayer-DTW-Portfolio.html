01 Project Overview

Why this project exists and what problem it solves.

🎯

The Problem

SAP Business One's legacy Data Transfer Workbench (DTW) relies on the DI API/COM stack, requiring a Windows client installation and complex setup. Modern integrations need a lightweight, HTTP-based alternative that works from any platform.

✅

The Solution

A fully REST-based import tool that speaks directly to the SAP B1 Service Layer. Upload CSV, TXT, or Excel files through a Blazor WebAssembly UI or REST API — no SAP client installation required. Validates, maps, and imports Business Partners with addresses, contacts, and user-defined fields.

02 Clean Architecture

The solution follows Clean Architecture principles with strict dependency rules — inner layers never reference outer layers.

Web UI  —  Blazor WebAssembly
Single-file upload page with import mode selection, error handling, and per-row result table. Communicates with the API exclusively through typed HttpClient services and shared DTOs.
▼   HTTP multipart/form-data   ▼
API Layer  —  ASP.NET Core Web API
Controllers: ImportController — POST /api/import/business-partners (upload file), POST /api/import/business-partners/export-log (download CSV log)
DI Composition Root: Registers MediatR, FluentValidation, file parsers, Service Layer HTTP client with cookie-based session management
API Docs: Scalar UI at /scalar/v1
▼   MediatR Command / Handler   ▼
Application Layer  —  Use Cases, Ports, Validation
Orchestrates the import pipeline: parses file → maps rows → validates → imports via Service Layer. Defines ports (interfaces) that the Infrastructure layer implements.
▼   Domain Models + Ports   ▼
Domain Layer  —  Entities, Enums, Value Objects
BusinessPartner, BPAddress, ContactPerson entities. CardType (Customer/Supplier/Lead), ImportMode (AddOnly/UpdateOnly/Upsert), ImportRowStatus (Success/Error/Skipped) enums. No external dependencies.
▼   HTTPS / REST   ▼
External  —  SAP Business One Service Layer
POST /b1s/v1/BusinessPartners (create)  |  PATCH /b1s/v1/BusinessPartners('{CardCode}') (update)  |  GET to check existence. Cookie-based session authentication (B1SESSION).

04 Key Features

What makes this tool powerful and production-ready.

📂 Multi-Format File Support

Accepts CSV (comma/semicolon), TXT (tab/delimiter), and Excel (.xlsx) files. A pluggable parser architecture (IFileParser + FileParserResolver) makes adding new formats trivial.

🧠 Smart Flat‑File Mapping

One flat row becomes a fully structured BusinessPartner domain object. Prefixed columns (BillTo_*, ShipTo_*, Contact{n}_*) automatically map to nested addresses and contact persons. U_* UDF columns pass through to SAP without code changes.

🔍 Per‑Row Validation

FluentValidation rules check CardCode (required, max 15), CardName (required, max 100), EmailAddress format, and AddressType values before any SAP call is made.

🔄 Three Import Modes

AddOnly — creates new records, skips existing CardCodes. UpdateOnly — patches existing records, skips new CardCodes. Upsert — checks existence first (GET), then POSTs or PATCHes accordingly.

🛡️ Error Handling Strategies

Stop on first error (abort the entire job) or continue on error (process all rows and collect failures). Every row gets a status (Success / Error / Skipped) with a descriptive message.

📥 Downloadable Result Log

After import, the per-row result table can be exported as a CSV file via the /export-log endpoint — useful for auditing and corrections.

🔐 SAP Service Layer Auth

Cookie-based session management via HttpClient with shared CookieContainer. Handles login, logout, self-signed certificates (dev), and error message extraction from SAP's JSON error format.

🧩 UDF Pass‑Through

Any column starting with U_ in the import file is automatically captured in the domain model's UdfFields dictionary and serialized into the SAP payload via [JsonExtensionData] — no code changes needed per UDF.

🧪 Comprehensive Test Coverage

Unit tests for the CSV parser, the FluentValidation rules, the BusinessPartnerMapper (UDF extraction, address mapping, contact extraction), and the MediatR handler (mocked parser, SL client, and validator).

Handler implementation (simplified)

C#// 1. Resolve the right parser (CSV, Excel, TXT) by file extension
var rows = await _parserResolver.Resolve(command.FileName)
    .ParseAsync(command.FileStream, command.FileName, ct);

// 2. Map each flat row → BusinessPartner domain entity
var businessPartners = rows.Select(_mapper.ToDomain).ToList();

// 3. Login to SAP Service Layer (cookie session)
await _slClient.LoginAsync(ct);

foreach (var bp in businessPartners)
{
    // 4. Validate before calling SAP
    var validation = await _validator.ValidateAsync(bp, ct);
    if (!validation.IsValid) { /* record error */ continue; }

    // 5. Upsert: check if CardCode exists → POST or PATCH
    bool exists = await _slClient.BusinessPartnerExistsAsync(bp.CardCode, ct);
    if (exists)
        await _slClient.UpdateBusinessPartnerAsync(bp, ct);    // PATCH
    else
        await _slClient.CreateBusinessPartnerAsync(bp, ct);   // POST
}
// 6. Logout & return per-row results
await _slClient.LogoutAsync(ct);

09 REST API

Endpoints, request/response formats, and dependency injection setup.

Method Endpoint Description
POST /api/import/business-partners Upload a flat CSV/TXT/XLSX file. Parameters: file, mode (AddOnly|UpdateOnly|Upsert), stopOnError (bool). Returns ImportJobResponse with per-row results.

10 Testing

Comprehensive unit test coverage across all layers.

🧪 Validator Tests

Tests that BusinessPartnerValidator rejects empty CardCode, invalid email addresses, and accepts valid business partners. Uses FluentAssertions for readable assertions.

🧪 Mapper Tests

Validates that U_* columns are extracted into UdfFields, BillTo_/ShipTo_ columns become BPAddress objects with correct AddressType, and Contact{n}_* columns become ContactPerson objects. Tests the real BusinessPartnerMapper directly.

🧪 Handler Tests

Mocks all dependencies (IFileParser, IServiceLayerClient, IValidator) to test the ImportBusinessPartnersHandler in isolation. Verifies success count, error count, and proper handling of Upsert mode.

🧪 CsvParser Tests

Tests that the CSV parser correctly parses header + data rows, supports .csv and .txt extensions, and rejects .xlsx files. Uses theory tests for parameterized inputs.

11 Key Takeaways

🏗️ Production-Ready Clean Architecture

Strict layer separation with dependency inversion. Domain layer has zero external dependencies. Application defines ports; Infrastructure implements them. API composes everything via DI.

🔌 Pluggable & Extensible

Add a new file format: implement IFileParser and register in DI. Add a new SAP object type: create a new MediatR command/handler. Add a new UDF: just add a U_ column to your import file — no code change needed.

🌐 Full-Stack .NET

From Blazor WebAssembly in the browser to ASP.NET Core on the server to SAP B1 Service Layer on the backend — the entire stack is C# / .NET 10.

✅ Test-Driven Quality

Every critical path — parsing, mapping, validation, handler orchestration — is covered by unit tests using xUnit + Moq + FluentAssertions.
